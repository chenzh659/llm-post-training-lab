# Dataset Card: Chinese E-commerce CS Instruction & Preference Dataset

## Dataset Summary

This dataset is a **synthetic + template-based** Chinese (zh-CN) customer-service (CS) corpus for large-language-model post-training. It covers common e-commerce support scenarios and is released in two complementary formats:

| Format | Purpose | Unit |
|--------|---------|------|
| **SFT (instruction)** | Supervised fine-tuning of a helpful CS assistant | Multi-turn `messages` dialogues |
| **Preference** | Preference / alignment training (DPO, ORPO, KTO, etc.) | `(prompt, chosen, rejected)` triples |

All samples are **programmatically generated** from scenario templates and seeded dialogue skeletons. No real user tickets, chat logs, or scraped private data are included.

Typical use cases:

- Domain SFT for Chinese e-commerce CS assistants
- Preference optimization (chosen vs. rejected reply quality)
- Evaluation of tone, policy adherence, and multi-turn consistency in zh-CN support chat

---

## Domain & Language

| Field | Value |
|-------|--------|
| **Domain** | 中文电商智能客服 (Chinese e-commerce intelligent customer service) |
| **Language** | `zh-CN` (Simplified Chinese) |
| **Register** | Professional, polite customer-support tone; occasional casual user turns |
| **Channel** | Text chat (IM-style multi-turn) |

### Scenario families (seeded)

Generation is organized around five scenario families. Each family has multiple templates with slots (product type, order status, logistics node, promotion rule, etc.):

1. **Product QA** — specs, compatibility, usage, authenticity, stock
2. **Logistics** — tracking, delays, address change, last-mile issues
3. **Refund / return** — eligibility, process, partial refund, exchange
4. **Complaint** — quality issues, mis-delivery, attitude escalation, compensation
5. **Promotion rules** — coupons, full-reduction, flash sale, stacking limits, expiry

Templates are combinable (e.g. delayed logistics + refund request) to increase diversity without introducing real PII.

## Sources

| Source | Description | PII / private data |
|--------|-------------|--------------------|
| **Programmatic templates** | Slot-filled dialogue scripts with controlled randomness (names as placeholders, order IDs as synthetic tokens) | None — synthetic only |
| **Seeded scenarios** | Hand-authored scenario trees for the five families above; used as seeds for expansion | None — authored for this lab |
| **Web / platform scrapes** | **Not used** | N/A |
| **Production CS logs** | **Not used** | N/A |

**Guarantee:** This dataset contains **no scraped private data** and no identifiable real-user content. Any names, phone numbers, addresses, or order IDs appearing in samples are **synthetic placeholders** generated by the pipeline.

## Schema

### SFT (instruction) — `messages` format

Each record is a multi-turn conversation suitable for chat-style SFT (ChatML / OpenAI messages / Hugging Face `messages`).

```json
{
  "id": "sft_000123",
  "scenario": "logistics",
  "sub_scenario": "delay_inquiry",
  "messages": [
    {"role": "system", "content": "你是一名专业的中文电商客服助手……"},
    {"role": "user", "content": "我的订单怎么还没到？"},
    {"role": "assistant", "content": "您好，很抱歉让您久等了。请您提供一下订单号，我帮您查询物流进度。"},
    {"role": "user", "content": "订单号是 SYN-2026-00421。"},
    {"role": "assistant", "content": "……"}
  ],
  "meta": {
    "num_turns": 4,
    "template_id": "logistics_delay_v2",
    "seed": 42
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Unique sample ID |
| `scenario` | string | One of: `product_qa`, `logistics`, `refund`, `complaint`, `promotion` |
| `sub_scenario` | string | Fine-grained template tag |
| `messages` | list[{role, content}] | Roles: `system` (optional), `user`, `assistant` |
| `meta` | object | Generation metadata (turns, template, seed) |

### Preference — `chosen` / `rejected` format

Each record pairs a shared conversation context with a preferred and a dispreferred assistant reply (single-turn preference over the last assistant act, or full-reply preference).

```json
{
  "id": "pref_000089",
  "scenario": "refund",
  "prompt": [
    {"role": "system", "content": "你是一名专业的中文电商客服助手……"},
    {"role": "user", "content": "七天无理由退货怎么操作？商品已拆封。"},
    {"role": "assistant", "content": "……（可选：前文客服轮次）"},
    {"role": "user", "content": "那还能退吗？"}
  ],
  "chosen": "可以理解您的顾虑。根据平台规则，……（准确、礼貌、可执行）",
  "rejected": "拆了就不能退，自己看规则。（生硬 / 错误 / 推诿）",
  "meta": {
    "preference_axis": "policy_accuracy+tone",
    "template_id": "refund_opened_item_v1",
    "seed": 17
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Unique pair ID |
| `scenario` | string | Same family taxonomy as SFT |
| `prompt` | list[{role, content}] | Context up to (and not including) the scored assistant reply |
| `chosen` | string | Higher-quality assistant response |
| `rejected` | string | Lower-quality alternative (tone failure, wrong policy, hallucination, etc.) |
| `meta.preference_axis` | string | Why chosen > rejected (e.g. `tone`, `policy_accuracy`, `completeness`, `safety`) |

**Preference axes used in generation:**

- Policy / rule accuracy  
- Tone & empathy  
- Completeness & actionability  
- Honesty under uncertainty (no fake tracking / fake compensation)  
- Boundary handling (escalate when needed; no over-promise)

## Size Targets

Default targets (overridable via config):

| Split product | Default target | Notes |
|---------------|----------------|--------|
| SFT dialogues | **~2,000** | Multi-turn; balanced across 5 scenario families |
| Preference pairs | **~800** | Independent of SFT count; may share scenario templates |

Configuration knobs (illustrative):

```yaml
# configs/dataset.yaml (example)
sft:
  target_size: 2000
  min_turns: 2
  max_turns: 8
  scenario_weights:
    product_qa: 0.22
    logistics: 0.22
    refund: 0.22
    complaint: 0.18
    promotion: 0.16

preference:
  target_size: 800
  axes: [policy_accuracy, tone, completeness, honesty, boundary]
```

Actual counts after cleaning may be slightly below targets; see retention fields below.

## Cleaning Pipeline & Retention

### Pipeline stages

1. **Generate** — Expand templates with slot fillers and RNG seeds.  
2. **Validate schema** — Required keys, roles, non-empty content, valid scenario tags.  
3. **Language / script check** — Predominantly zh-CN; reject empty or garbage strings.  
4. **Deduplicate** — Near-duplicate detection on normalized user+assistant text (hash / MinHash).  
5. **Quality filters** — Minimum length, max repetition, no unresolved slots (`{{...}}`), no leaked system-prompt-only samples.  
6. **Preference consistency** — Ensure `chosen != rejected`; optional length / axis tag presence.  
7. **Split** — Stratified by `scenario` into train / val / test.  
8. **Export** — JSONL for SFT and preference; write cleaning report.

### Expected retention & report fields

Cleaning writes a report (e.g. `reports/cleaning_stats.json`) with fields such as:

| Field | Meaning |
|-------|---------|
| `generated_raw` | Count before filtering |
| `after_schema` | Passed schema validation |
| `after_language` | Passed language checks |
| `after_dedup` | Unique after near-dedup |
| `after_quality` | Passed length / slot / repetition filters |
| `final_export` | Written to split files |
| `retention_rate` | `final_export / generated_raw` |
| `drop_reasons` | Histogram of rejection causes |
| `per_scenario` | Counts and retention by scenario family |

**Expected retention rate (order of magnitude):** ~70–90% of raw generated samples, depending on template diversity and dedup aggressiveness. Prefer oversampling generation (`target_size / expected_retention`) so the **final** export meets the size targets above.

## Split Ratios

All exports use a stratified **8 : 1 : 1** split by `scenario` (and seed for reproducibility):

| Split | Ratio | Role |
|-------|-------|------|
| **train** | 0.8 | SFT / preference training |
| **val** | 0.1 | Hyperparameter selection, early stopping |
| **test** | 0.1 | Held-out evaluation only |

Recommended layout:

```text
data/
  sft/
    train.jsonl
    val.jsonl
    test.jsonl
  preference/
    train.jsonl
    val.jsonl
    test.jsonl
  reports/
    cleaning_stats.json
```

Do not train on `test`. Prefer fixing the random seed in config for bit-stable regenerations when template code is unchanged.

## Ethical Considerations

- **No private data:** Samples are synthetic/template-only. Do not mix in production CS logs without a separate legal review, redaction pipeline, and updated dataset card.  
- **Synthetic risk:** Models may still overfit template phrasing; treat metrics on this set as **in-domain synthetic** performance, not a substitute for real-user evaluation.  
- **Policy fiction:** Promotion and refund “rules” in templates are **lab-defined** and may not match any real platform. Downstream products must replace them with official policy sources.  
- **Bias & tone:** Templates aim for professional, non-discriminatory language. Generators should avoid stereotypes related to gender, region, or occupation of customers.  
- **Safety:** Complaint/escalation templates should not teach harmful advice, doxxing, or fraudulent refund coaching. Rejected preference samples may intentionally show bad tone for training contrast—they are not recommended as system behavior.  
- **Attribution:** If regenerating for publication, document generator version, config hash, and seed so others can reproduce or audit the corpus.  
- **Intended use:** Research and portfolio / post-training lab demos. Not a certified compliance corpus for regulated consumer-protection claims.

## How to Regenerate

From the project root (after implementing the generator scripts described in the lab README):

```bash
# 1) Install dependencies (example)
pip install -r requirements.txt

# 2) Edit size / weights / seed
#    configs/dataset.yaml  → sft.target_size, preference.target_size, seed

# 3) Generate raw SFT + preference
python scripts/generate_dataset.py --config configs/dataset.yaml --out data/raw

# 4) Clean, dedup, split 8:1:1
python scripts/clean_and_split.py --in data/raw --out data --ratio 0.8,0.1,0.1

# 5) Inspect retention
python scripts/report_cleaning.py --stats data/reports/cleaning_stats.json
```

**Reproducibility checklist:**

1. Pin generator commit / version and `configs/dataset.yaml`.  
2. Set a fixed global `seed`.  
3. Record `generated_raw`, `final_export`, and `retention_rate` in the cleaning report.  
4. Version the export directory (e.g. `data/v1/`) when templates change.

If only sizes change, keep templates fixed and re-run generate → clean → split; update this card’s **Size Targets** section to match the new config.
